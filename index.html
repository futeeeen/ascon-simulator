解密 ASCON 演算法

為物聯網與資源受限裝置而生的新世代輕量級密碼學標準

📦 ASCON 家族核心成員

ASCON 並非單一演算法，而是一個密碼學套件家族，對應不同的安全需求：

Ascon-AEAD128

主打： 認證加密 (Authenticated Encryption)

規格： 128-bit 金鑰與安全強度。

特色： 同時保障資料的「機密性」與「完整性」，是物聯網通訊防護的首選。

Ascon-Hash256

主打： 資料完整性 (Hashing)

規格： 產生 256-bit 的固定長度輸出。

特色： 適用於韌體更新驗證、數位簽章等不需解密的場景。

💡 核心共通點筆記：
無論是 AEAD 還是 Hash，它們底層都共用了極其相似的海綿結構 (Sponge Construction) 與 320-bit 核心排列函數 (Ascon-p)。這意味著在晶片硬體實作時，這兩種功能可以「共用同一套電路」，大幅節省硬體面積與耗電量！

🔄 ASCON-AEAD 加密流程解析

ASCON-AEAD 的運作就像是一條極具效率的輸送帶，資料只需通過一次 (Single-pass) 即可完成加密與認證：

1. Initialization (初始化)

將 Key (金鑰)、Nonce (絕對不重複的隨機數) 與 IV 寫入 320-bit 的內部狀態中。接著透過排列函數劇烈攪拌，建立不可預測的初始防護罩。

2. Associated Data (關聯資料吸收)

處理封包表頭 (Header) 等「不需加密、但防竄改」的資料。將其 XOR 融入狀態中，最後加入一個 Domain Separation Bit，用來明確區分接下來的階段。

3. Plaintext / Ciphertext (明文處理)

將機密的 Plaintext (明文) 與內部狀態進行 XOR 運算，直接產生 Ciphertext (密文)。每處理一個區塊，就再次呼叫排列函數更新狀態，確保前後文關聯。

4. Finalization (最終化)

再次將 Key (金鑰) 注入狀態中做最後的強烈攪拌，並提取出 128-bit 的 Tag (認證標籤)。這個 Tag 就是整份訊息的數位防偽印章。

🛡️ 鐵壁般的安全性機制

解密時，驗證失敗絕對不能輸出明文！

在 ASCON 的解密過程中，系統會先計算出標籤，並與夾帶的 Tag 進行比對。如果 Tag 不符，系統只會無情地回傳 `fail`，絕對不會交出任何解密到一半的明文。

🧠 秒懂比喻：毒藥膠囊與機密保險箱

想像一個裝有機密文件的智慧保險箱。開啟它的條件是必須核對一組「超長指紋特徵 (Tag)」。

如果今天駭客隨便輸入錯誤的指紋，保險箱如果選擇「門開一條縫，讓你看一眼裡面模糊的文件，然後再關上說密碼錯誤」，駭客就可以透過無數次故意輸入錯誤指紋，觀察那「一條縫」裡的變化，慢慢拼湊出完整機密（這在密碼學中稱為 Padding Oracle Attack 選擇密文攻擊）。

因此，ASCON 的設計就像是內部裝了毒藥膠囊的保險箱：只要指紋 (Tag) 不對，裡面的文件瞬間銷毀，連一點紙屑都不留給你。100% 驗證通過才給看，是保護機密的第一鐵則！
